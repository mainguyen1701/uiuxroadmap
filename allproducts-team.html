Human-centred design — who we are designing for
Consistency is the foundation. Empathy is the ceiling.
Phases 0–3 make Applicaa consistent. But consistency alone does not make software feel human. The second dimension of this roadmap is designing for the emotional reality of the people using it — a school admin under pressure, a parent anxious about their child's future. Understanding that emotional state is what separates a product that works from one people trust.
Who we design for — empathy maps (NN/G framework: Says · Thinks · Does · Feels)
The school admissions coordinator
Primary user — Admissions+ and MAT+
Says
"Where is that setting again?" · "Why is this import failing?" · "I need this done before 3pm."
Thinks
I'm responsible if this goes wrong. I shouldn't need to call support for this. This should be faster.
Does
Screenshots settings to remember them. Keeps a spreadsheet alongside the product. Calls colleagues for help navigating the UI.
Feels
Time pressure. Accountability without clarity. Relief when things save. Frustration when errors give no explanation.
The parent applying for their child
End user — Admissions+ application flow (Phase 2+)
Says
"Did it submit?" · "What do I do next?" · "I don't have that document right now."
Thinks
This is my child's future. What if I make a mistake? Is this safe? Will someone actually read this?
Does
Reads everything twice before clicking. Abandons and returns. Takes screenshots of confirmation. Calls the school to check it arrived.
Feels
Anxiety about consequences. Uncertainty about the process. Relief at clear confirmation. Trust built through transparency.
Four trust factors — stable since 1999, still true today (NN/G)
01
Design quality signals legitimacy
A typo on a homepage, misaligned components, or inconsistent colours all communicate the same thing: lack of attention to detail. Visual quality is the first trust signal — before a user reads a single word.
02
Upfront disclosure builds confidence
Hidden information destroys trust the moment it is discovered. Show import status. Explain what happens after an offer is made. Tell the parent exactly where their application is. No surprises.
03
Accurate, current content
Error messages that say nothing. Status badges that mean different things on different screens. Loading states with no explanation. Each one erodes trust. Accurate, current feedback is not a nice-to-have — it is the minimum.
04
Compassion, not sympathy
Sympathy acknowledges a user's problem. Compassion compels you to fix it. The goal is not longer error messages apologising for failures — it is designing so the failure does not occur, and when it does, recovery is instant.

Risks and Mitigations
Three risks the team has identified and how each is managed.
⏱️
Timeline risk
Phase 2 and Phase 3 run in parallel in March–April. If research synthesis or engineering availability slips, the April milestone moves.
Mitigation
AI-accelerated prototyping (v0, Cursor, Claude) compresses Phase 2 from months to weeks. Phase 2 scope is prototypes only — not builds. If one product track slips, the other ships independently.
🔧
Engineering dependency
Phases 1–3 require engineering capacity to implement validated designs. If engineering is unavailable or reprioritised, shipped designs may lag.
Mitigation
Designers build HTML and Tailwind prototypes directly — reducing engineering effort on each item. Phase 1 items are independently shippable, so the team can flex around engineering capacity without losing the whole phase.
🤖
AI capability uncertainty
Phase 4 AI features depend on engineering building the underlying models on schedule (AR 2 roadmap). If AI capabilities ship later, Phase 4 UX design stages shift accordingly.
Mitigation
Phase 4 UX design leads engineering — shared AI DS components, confidence patterns, and escalation flows are designed and in the Lookbook before any AI feature ships. Design is never the blocker.

What is the DS Lookbook?
The DS Lookbook at designsaplus.com/lookbook is the shared design system for Admissions+ and MAT+ — every reusable component, pattern, and token in one place. When two screens look different, the Lookbook is the arbiter of which is correct.

What Phase 0 delivers
🧱
Component library
Admissions+ MAT+
Every reusable UI element — inputs, buttons, tables, cards, panels, navigation, badges, and feedback components — defined in one place with a live preview, available variants, and usage guidance. No more ad-hoc rebuilds or searching for the right example.
Every developer and designer works from the same reference. Components look and behave consistently across every screen.
🎨
Design tokens
Admissions+ MAT+
The colour palette, typography scale, spacing system, and status colour map — all named and published as shared tokens. Navy #263C91, Coral #EC634B, status colours (Upcoming = blue, Completed = green, Cancelled = red, Full = amber), and the full Inter type scale are all defined here.
Colour, type, and spacing are consistent across the product. A future rebrand or dark mode is a token change — not a product-wide rebuild.
📐
Shared patterns
Admissions+ MAT+
Common interaction patterns documented beyond individual components — how filtering works across list screens, how empty and loading states are handled, how record detail panels open and close, how status is communicated consistently. These are the decisions that, without a shared reference, each developer makes differently on every screen.
Staff experience the same behaviours on every screen. Learning one list means understanding all lists.

📐 Design team reference — background reading
Research foundation — Nielsen Norman Group, 2025
AI is the third UI paradigm in 60 years. Conversational design is not a feature — it is a new interaction model.
For 60 years, interfaces were command-based. AI introduces intent-based outcome specification — users state what they want, the system works out how. This is not a chat window on existing UI. It requires designing outcomes and constraints, not screens and flows.
1945 – 1964
Batch processing
Users submitted complete instructions and waited for output. No feedback. No adjustment. One shot.
1964 – 2022
Command-based GUI
Users issue commands, see results, adjust. This is the model all Admissions+ screens currently follow — including every screen in this roadmap.
2022 onwards — Applicaa Phase 4
Intent-based outcome specification
Users state what they want. The system interprets intent, takes action, and surfaces a result. The designer's job shifts from designing steps to designing outcomes — and the guardrails the AI must operate within.
Six conversational design principles guiding Phase 4
Be transparent about being AI
NN/G research shows users calibrate language and expectations when they know they are talking to AI. Hiding it destroys trust the moment they discover it. Every Applicaa AI interaction identifies itself and its confidence level upfront.
Offer structure, allow flexibility
Prepopulated options for common inputs (buttons, chips, suggested answers) reduce typing effort. Free text input allows deviation. Both must coexist — forcing users to type everything or click everything both fail. Design for the middle.
Progressive disclosure of complexity
Reveal complexity gradually. Start with what the user said they want. Show detail only when they need it. Do not present every option upfront — the AI should ask one question at a time and offer depth on demand.
Design for deviation
Chatbots fail when users deviate from linear flows (NN/G). Every conversational flow must have a designed recovery path — what happens when the AI does not understand, when the user changes direction, or when the answer is not in the expected format.
Outcome-oriented constraints over screen design
In generative UI, designers define what the AI must achieve and the guardrails it must operate within — not every screen element. The designer specifies the goal, the constraints, and the fallback. The AI generates the interface moment by moment.
Speed as the trust signal
NN/G research: the primary advantage of AI over human agents is speed. Every interaction should be designed to reach resolution faster than the current screen-based equivalent. If it is not faster, the AI has not earned its place in the product.
Sources: Nielsen Norman Group — "AI: First New UI Paradigm in 60 Years" and "Generative UI and Outcome-Oriented Design" (2025)
Designing with AI for trust and empathy
AI cannot feel empathy. But the experience of being understood can be designed.
Five principles shaping every AI interaction — from a family opening an application to a coordinator approving a recommendation.
1
Empathy is designed, not programmed
The feeling of being understood does not require the AI to understand. It requires the designer to understand — and encode that understanding into every interaction, state, and moment of friction.
How to execute
Run empathy mapping before designing any AI flow. Fill the Thinks quadrant first — what is the user afraid to say out loud? For a family filling an application: "Am I doing this wrong? Will my child miss out because of me?" That fear must shape every micro-copy choice.
Design reassurance into the idle state. If a user pauses on a question for more than 30 seconds, the AI offers help proactively — not with an apology, with a shortcut: "Most families complete this by uploading their last school report. Want to try that?"
Use progress signals that feel human. Not "Step 3 of 8" — "You're about halfway. The next section is short." Language that acknowledges effort, not just position.
Design for the person behind the task. A school admin processing offers at 7pm is tired and time-pressured. The AI surfaces the three most urgent actions first — unprompted. That is designed empathy.
2
Trust with AI forms faster and breaks faster than with humans
NN/G research: users form their trust model of an AI in the first 3 interactions. A confident wrong answer in interaction 1 can end the relationship entirely. The designer's job is to earn trust early and protect it deliberately.
How to execute
Design the first interaction as a trust demo, not a task. The Admissions Copilot's first action should be something impressive and low-risk — "I've pre-filled your school name and year group from last year's data. Does that look right?" shows capability without asking for anything.
Never let AI be confident and wrong. Every AI output that could be incorrect must show a confidence indicator. If the AI is below 80% confident, it asks rather than states. "I think the deadline is 14 March — is that right?" is trustworthy. "The deadline is 14 March." followed by a correction is trust-breaking.
Celebrate early wins visibly. When the AI correctly reads a document, summarises an application, or catches an anomaly — surface that success explicitly: "I spotted something you might want to check." This is the AI showing its value, not claiming it.
Design the recovery moment. When AI gets something wrong, the recovery interaction is more important than the error itself. One-tap correction, plain acknowledgement ("Thanks — I'll remember that"), and no repeat of the same mistake in the same session.
Trust-breaking
"The deadline is 14 March."
"I'm sorry, I made an error."
Confident answer with no source
Repeats the same mistake
Trust-building
"I think it's 14 March — can you confirm?"
"Got it, corrected. What should I do next time?"
Confidence level shown next to every answer
Adapts within the same session
3
Users don't reach for AI even when it can help
NN/G diary study: only 7% of needs were addressed by AI — even though 62% could have been. The gap is not capability. It is visibility and trust. Users don't know what the AI can do, or they don't believe it will work until they've seen it work.
How to execute
Make AI capabilities visible at the moment of need. When a staff member is on the application list page, the Copilot surface shows: "Ask me anything about these applications." Not buried in a menu — contextual, present, specific to the current screen.
Use suggested prompts to show possibility. The natural language query interface should pre-populate 3 example questions relevant to the current screen: "How many applications are missing a reference?" "Which applicants haven't RSVPd to the open day?" Users discover capability through examples, not documentation.
Design the first-use moment deliberately. The first time any staff member sees the Copilot, it should proactively do something useful — without being asked. Show don't tell. An unprompted insight lands harder than any onboarding screen.
Track and surface usage. If a coordinator has never used natural language queries, the system can surface: "Did you know you can ask me questions about your pipeline?" — once, at the right moment, not as a persistent badge.
4
Empathy mapping surfaces what users will never put in a support ticket
NN/G: the Thinks quadrant captures what users are feeling but won't say out loud — self-doubt, fear, anxiety. For Applicaa users, those feelings are hidden but consequential. Designing for them is the difference between a product that works and one that actually helps.
How to execute
Run an empathy mapping session before Phase 2 redesign begins for each product. For Admissions+: "A parent filling in an application form for the first time." For MAT+: "A trust coordinator preparing the board report the night before it's due." Map Says / Thinks / Does / Feels — spend 80% of the session on Thinks.
Design for the unspoken anxiety of the application form. Families fear: making a mistake that affects their child's place. Every save confirmation, every validation message, and every error should speak to that fear directly: "Your application is saved automatically — you won't lose anything." Not just a toast notification.
Design for the unspoken anxiety of the AI recommendation. Staff fear: being held responsible for a decision the AI made. The approval interface must make the human feel like they genuinely made the call — not just rubber-stamped a machine. Show reasoning, not just recommendation.
Add a Thinks column to the Phase 2 design brief for every screen. Before designing any screen, document: "What is the user afraid of at this moment?" That answer shapes the micro-copy, the state design, and the error handling — not just the layout.
5
Empower, don't apologise
NN/G: "The goal of design is not to be nice to users, but to empower them." Sympathy says sorry. Empathy fixes the problem. Every AI failure mode, error state, and recovery moment must be designed to move the user forward — not to make the product feel better about itself.
How to execute
Rewrite every AI error state as an action. "I couldn't understand that" becomes "Here are two ways I can help — which is closer to what you meant?" The user is never left with a dead end. Every failure surfaces a path forward.
Replace apology language with action language across all phases. Phase 1 import errors: not "Something went wrong with your file" — "Row 14 has a missing email address. Fix it and re-upload, or skip that row." Phase 4 AI: not "I'm sorry, I made a mistake" — "I got that wrong — here's the correct answer."
Design the override as empowerment, not correction. When a staff member overrides an AI recommendation, the interface should feel like the human won — not that the AI failed. "Your call — updated." And offer: "Should I factor this in next time?" That is the AI learning, not the AI apologising.
Audit every error message across Phases 1–4 against one question: does this move the user forward, or does it just describe what went wrong? Every message that only describes the problem is a sympathy response. Every message that provides a next action is an empathy response.
