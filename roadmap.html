Design System Foundation

Fix the shared foundation and build domain-specific custom components. Nothing in Phases 1–5 can be consistent without this. The DS Lookbook is the delivery tool — each component ships into the product, not just the preview.

12
items
4
P0
Fix — Repair before building
Publish 4 broken Lookbook previews
P0MAll
Problem: DataTable, TagInput, InfoGrid, PersonCard exist in code but Lookbook returns 404. Teams rebuild these ad-hoc on every screen.
Write Lookbook previews for all 4 with props, variants, real ApplicaaOne data examples, and usage notes. Block any new usage until documented.
Remove Bootstrap class leakage
P0SAll
Problem: fw-medium, fw-bold, text-muted, text-danger scattered across 20+ component templates. Break silently if Bootstrap is removed.
Find-replace across all ViewComponent templates: text-muted → tw:text-neutral-400, fw-bold → tw:font-bold, text-danger → tw:text-red-600. Add linter rule to prevent reintroduction.
Fix semantic badge colours
P0SAll
Problem: "Upcoming" events use badge-success-subtle (green). badge-info-subtle already exists in the Enquiries DS — just not applied consistently in MAT+.
Audit badge usage across all 3 products. Enforce: Upcoming → info, Completed → success, Cancelled → danger, Full → warning, Draft → neutral. Document the map in the DS.
Vanilla JS → Stimulus (Task reminders)
P1MAdmissions+
Problem: Enquiry record Add Task popup uses onclick="addDateReminder()" — raw JS bypassing the Stimulus architecture. Untestable, inconsistent, can't be reused.
Replace with reminders_controller using Stimulus targets and actions. Reminder rows become a Lookbook-documented component with add/remove lifecycle.
Build — Custom domain-specific components
Empty State component
P0MAll
Problem: Every table and panel handles zero-data inline. No consistent pattern — three different ad-hoc styles across 3 products.
Build tw-empty-state: icon/illustration slot, heading, body, optional CTA. Variants: no-data, no-results (after filtering), first-use (onboarding prompt with action).
Skeleton Loader
P0MAll
Problem: Every async fetch (events table, RSVP list, Bromcom import, KPI dashboard) shows nothing while loading. Layout shift, feels broken.
Build tw-skeleton with CSS shimmer animation. Variants: text line, avatar, card, table row, stat block. Zero JS dependency — pure CSS keyframe.
Filter Chips component
P0MAll
Problem: Push to Schools, enquiry list, timeline filters, year group filters — all bespoke Alpine.js. Same interaction built 6+ times with no shared component.
Build tw-filter-chip + tw-filter-chip-group. States: default, active, disabled. Optional count badge, "All" default chip, clear-all shortcut. Stimulus-powered.
Date & Time Picker
P0LAll
Problem: Log Call, Add Task, event scheduling — all use <input type="text"> or double <select> for time. Inconsistent format, no validation, browser-dependent behaviour.
Build tw-date-picker (single, range) and tw-time-picker. Keyboard accessible, form-integrated, DD/MM/YYYY locale. Replaces all date/time inputs across all 3 products.
Page-level System Banner
P1MAll
Problem: No component for platform-wide persistent alerts: Bromcom import running, subscription expiring, MIS sync failure, maintenance window. Currently unhandled across all 3 products.
Build tw-banner above topbar. Variants: info, warning, critical. Dismissable, server-managed via Rails flash or a Banner model. Stacks if multiple active.
Column table density — reduce to 7 default
P0MMAT+
Problem: Events table has 15 columns — overflows on laptops, unusable on iPads. Primary device for UK school staff is iPad/Android (BYOD).
Default: 7 columns (Actions, Date, Event, RSVPs/Cap, Status, Source, Tickets Remaining). Add column chooser using existing view_toggle_controller. Move QR to action dropdown.
tw-person-card footer slot
P2SAdmissions+
Problem: tw-person-card-footer class exists in the DS but is never shown in the Lookbook. No documented pattern for action buttons (Call, Email, View profile) on a person card.
Add footer slot to Lookbook preview with real actions. Document: primary action (View record), secondary actions (Call, Email). Align with enquiry record parent card usage.
SMS stats + character counter
P2MAdmissions+
Problem: SMS popup shows "Messages remaining / Recipients / UK phone" as bare <p> tags. Character counter is raw text. Both patterns repeat — no DS component.
Build tw-character-counter (attaches to textarea, shows N/max, warning colour at 80%). Build tw-sms-stat-row for the send stats block.

Quick Wins — All Products

High-impact, low-effort fixes that improve the daily experience for school staff users and shift platform perception. Every item ships independently — no cross-dependency. Focus: settings pages, recurring processes, filters, and navigation.

12
items
7
P1
Settings pages — the most-neglected daily surface
Settings page consistency audit
P1MAll
Problem: Settings pages across all 3 products use different layouts, form patterns, and save behaviour. Staff who use all three products hit a different experience for the same task.
Standardise: left nav for categories, tw-form-section + tw-form-group layout, sticky footer with Save/Cancel, inline success toast on save. Document as "Settings Page" pattern in DS.
Permissions & access settings — clarity
P1MMAT+
Problem: Staff permissions in MAT+ live inside a modal — cramped, hard to read cross-school access, no legend for what each level can do.
Move to a dedicated Permissions settings screen. School-grouped staff table, access level legend, add staff via searchable combobox, bulk-save changes. Use tw-info-grid for the legend.
Processes — recurring workflows staff do every day
Bromcom import — progress & error feedback
P1MMAT+Admissions+
Problem: Data imports run silently. Staff don't know if it succeeded, is running, or failed until they navigate away and back. High anxiety action with no feedback loop.
Add tw-progress (indeterminate while running), success/fail tw-inline-notification, error rows in a diff-style table. Use Turbo Streams for real-time updates — no page reload.
Offer management — status clarity
P1MAdmissions+
Problem: Making and managing offers — conditional, unconditional, waitlisted, declined — involves dense rows with no visual differentiation between offer states. Everything looks the same.
Status-coloured badge per row. Top KPI row (offers made/accepted/declined) using tw-info-grid-inline. Bulk offer actions. Offer state change triggers timeline entry automatically.
Event booking — push workflow
P1SMAT+
Problem: Push to Schools modal has 15 schools in a flat list with no phase filter chips, no "select all visible" for a filtered subset, and no per-school date configuration surfaced clearly.
Apply tw-filter-chip (built in Phase 0) for phase filtering. "Select All Visible" respects active filter. Per-school date toggle in the same step — no separate modal needed.
Enquiry record — log call & task UX
P1SAdmissions+
Problem: Log Call and Add Task use plain text inputs for dates and double-selects for time. After Phase 0 builds the date/time picker, these need to be wired up.
Replace all date/time fields in Log Call, Add Task, and Send SMS with tw-date-picker and tw-time-picker. Wire task reminders to reminders_controller (built in Phase 0).
Filters — the most-used interaction in every list view
All Enquiries list — filter chips
P1MAdmissions+
Problem: The allenquiries Lookbook scenario is blank — no DS reference exists for the main enquiry list view (filter chips, bulk bar, empty state).
Build the All Enquiries Lookbook scenario using tw-filter-chip (stage, source, date range). Add tw-data-table with tw-empty-state for zero results. Publish and ship.
Timeline activity type filters
P1SAdmissions+
Problem: Enquiry record timeline has type filter buttons (All, Emails, SMS, Calls, Notes, Tasks, Events) as bespoke inline buttons — not using tw-filter-chip.
Replace with tw-filter-chip-group. One-line change once the component exists from Phase 0. Active state, count badges per type.
Navigation — dual audience split
P0MApplicaaOne
Problem: ApplicaaOne nav doesn't separate student-facing from school-facing journeys. A Year 10 student and an admissions officer hit the same nav — different needs, one confusing structure.
Split nav: "For Students" and "For Schools" as distinct paths. Persistent course search for students. Mobile-first hamburger with clear back-navigation.
Colour token consolidation
P0SApplicaaOne
Problem: 7 blue variants in use: text-primary-5, text-brand-blue, text-blue-600, bg-[#2D8FC5], bg-blue-500 and more. Impossible to maintain.
Consolidate to 3 semantic tokens: brand-primary (#263C91), brand-secondary (#2D8FC5), brand-dark (#1E2F72). Alias all existing classes. Document as colour reference in DS.

Core Component Build

Build the missing components that gate Phase 3 product redesigns. Without these, every new screen falls back to bespoke implementations that diverge from the system.

7
components
5
P1
Detail Slide-out Panel
P1LAll
Problem: RSVP panel, school detail, student record — all slide in from the right, all built custom. The nav Drawer is a navigation component, not a record detail panel.
tw-panel: right-side slide-in, sticky header, scrollable body, sticky footer. Sizes: sm (480px), md (600px), lg (800px). Backdrop lock, focus trap, close on Esc.
Bulk Action Bar
P1MAll
Problem: Every table with multi-select re-implements the selection toolbar (X items selected | Publish | Cancel | Delete). No shared component — inconsistent placement and behaviour.
tw-bulk-bar: contextual bar above the table on row selection. Slots: count, primary action, secondary actions, deselect-all. Connects to filter_controller.
Stat / KPI Card
P1MAll
Problem: tw-info-grid exists in Enquiries for key/value pairs but no metric card component exists for KPI dashboards (MAT+ overview, Admissions+ reports, Insights tab).
tw-stat-card: primary metric, label, delta indicator (▲▼ + colour), comparison text ("vs last month"), sparkline slot. Works standalone and in a responsive grid.
Combobox / Multi-select
P1LAll
Problem: School selection, year group, staff assignment, subject filter — all need searchable multi-select. Existing Dropdown and FormSelect don't support search + constrained multi.
tw-combobox: keyboard-navigable, single + multi mode, option groups, "select all", chip removal. Stimulus-powered, no external library dependency.
Calendar View
P1XLMAT+Admissions+
Problem: Events calendar in MAT+ is entirely custom Alpine.js — can't be reused for Admissions+ event bookings without full re-implementation.
tw-calendar: month + week toggle, event pill per cell, colour by type, today highlight, click-to-open popup hook. Data via JSON attribute. Stimulus controller.
Avatar Group
P2SAll
Problem: Event attendees, staff lists, team displays all show count text with no visual identity. Max-3 overlapping avatars is a standard pattern the DS doesn't have.
tw-avatar-group extending tw-avatar. Max visible configurable (default 3). Overflow: "+N more" with tooltip listing full set.
Split Button
P2MMAT+
Problem: Publish, Push to Schools, Make Offer — all have a primary action + secondary options currently shown as separate Button + Dropdown pairs with no shared housing.
tw-split-btn: primary area + chevron dropdown as distinct click targets, same height and border. Variants: primary, success. Inherits dropdown_controller.

Scale & Polish

Platform-wide quality pass: accessibility, mobile, animation, and documentation. These ensure the revamp holds long-term and the DS is maintainable by the full team without supervision.

5
items
WCAG 2.1 AA accessibility audit
P0LAll
Problem: UK public sector procurement requires WCAG 2.1 AA. Schools and MATs are public-sector — Applicaa handling student data must meet accessibility standards or risk losing contracts.
Audit all 3 products: colour contrast (4.5:1 text, 3:1 UI), keyboard nav, focus management, screen reader labels, skip links, form error announcements. Fix critical findings. Publish accessibility statement.
Mobile-first responsive pass — product
P1LAll
Problem: UK schools use iPads and Android tablets (BYOD). Core screens were designed desktop-first. Phase 1 covered Admissions+ flows — this pass covers MAT+ and ApplicaaOne.
Top 10 views per product on 375px and 768px. Convert remaining desktop-only tables to card layout on mobile. Real device testing on iOS Safari + Android Chrome.
Animation & motion tokens
P2MAll
Problem: Transitions between states (modal open, panel slide, skeleton-to-content, toast) are inconsistent — some Tailwind, some Stimulus, some CSS keyframes. No shared motion language.
Define 4 motion tokens: duration-fast (100ms), duration-base (200ms), duration-slow (350ms), easing-standard (ease-in-out). Respect prefers-reduced-motion.
DS contribution guide & governance
P1MAll
Problem: No documented contribution process. Any developer can add a component in any style — which is how Bootstrap leakage and semantic colour misuse happened in the first place.
Publish in Lookbook: component API contract, naming conventions, required scenarios, accessibility checklist, PR review criteria. Add automated linter for Bootstrap remnant classes.
Dark mode token preparation
P2MAll
Problem: No dark mode. Admissions staff work evenings. Parents register late at night. School events happen at 7pm. Dark mode expectation is growing on all personal devices.
Define semantic colour tokens: surface-primary, text-primary, border-default. Map to light values now. Dark values prepared but not shipped. No UI toggle in this phase.

AI-Native Product Evolution

Redesign interactions for AI-assisted workflows — conversational UX, trust-building patterns, confidence indicators, graceful error recovery, and agent-to-human handoffs. Applies across Admissions+, ApplicaaOne, and MAT+ as AI features ship.

5
tracks
4
projects
Trust-building patterns — designing for AI uncertainty
💬
Conversational UX Framework
Replace form-heavy flows with natural conversation for parents. Design the trust arc:
  • Introduction — "I'm here to help with John's application"
  • Capability demo — "Try uploading his transcript"
  • Collaborative — "I notice his math grades meet requirements"
  • Handoff — "Ready for Ms. Smith to review"
📊
Progressive Trust Design
Design confidence indicators for AI recommendations across admissions decisions:
90%+ · "I'm confident this is correct"
60–89% · "Let me double-check with you"
<60% · "I need help — can you clarify?"
🔄
Error Recovery & Escalation
Design graceful failure modes for every AI error type:
  • Misunderstood input → suggested rephrase
  • Missing information → explain what's needed and why
  • System limitation → clear escalation to human
  • Data mismatch → show exactly what's wrong
Escalation path: Agent struggle → Suggest clarification → Human assistance → Direct contact
Implementation projects
Conversation Flow Redesign
P1LAdmissions+
Goal: Replace enquiry and application forms with guided conversation — 90%+ completion rate, <5 clarification requests per application.
Script 5 common scenarios. Design branching conversation trees. Build Figma prototypes with realistic content. Test with real parents including voice interface. Build error state library for all AI failure modes.
Trust Dashboard for School Staff
P1LMAT+Admissions+
Goal: Help staff understand and trust AI recommendations on applications, shortlisting, and offer decisions.
Confidence visualisation per recommendation. Reasoning display ("Why I suggested accept"). Override interface — easy disagree + teach-AI loop. Full audit trail of all AI decisions. tw-stat-card for AI accuracy metrics.
Mobile-First Parent AI Experience
P1LApplicaaOne
Goal: 80%+ of parents apply via mobile. AI conversation must work on small screens with voice input and intermittent connectivity.
Optimise chat bubbles and input for small screens. Camera integration for document upload. Push notifications for status updates. Voice input mode. Offline-capable save state. Real-time translation for 20+ languages.
AI Design System Patterns
P2MAll
Goal: Create reusable AI-specific DS components so every team building AI features uses consistent patterns.
tw-chat-bubble (inbound/outbound, read receipt), tw-confidence-indicator (high/mid/low), tw-ai-thinking (loading state for AI responses), tw-escalation-card (human handoff). Publish all to Lookbook with AI-specific usage guidelines.
