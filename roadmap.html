Phase 0 — Shared Foundation

Establish the Admissions+ design system using the DS Lookbook — the single shared reference for how every component, pattern, colour, and interaction should look and behave. Every improvement in Phases 1–4 is built using this language. Without it, each screen is built from scratch and drifts from everything else.

What is the DS Lookbook?
The DS Lookbook at designsaplus.com/lookbook is the Admissions+ design system — a living reference that documents every reusable component, shared pattern, and design token used across the product. It is the source of truth for how things look and behave.

When a developer needs to build a new screen, they start here. When a designer proposes a change, they document it here first. When two screens look different, the Lookbook is the arbiter of which is correct.
What Phase 0 delivers
🧱
Component library
Admissions+
Every reusable UI element — inputs, buttons, tables, cards, panels, navigation, badges, and feedback components — defined in one place with a live preview, available variants, and usage guidance. No more ad-hoc rebuilds or searching for the right example.
Every developer and designer works from the same reference. Components look and behave consistently across every screen.
🎨
Design tokens
Admissions+
The colour palette, typography scale, spacing system, and status colour map — all named and published as shared tokens. Navy #263C91, Coral #EC634B, status colours (Upcoming = blue, Completed = green, Cancelled = red, Full = amber), and the full Inter type scale are all defined here.
Colour, type, and spacing are consistent across the product. A future rebrand or dark mode is a token change — not a product-wide rebuild.
📐
Shared patterns
Admissions+
Common interaction patterns documented beyond individual components — how filtering works across list screens, how empty and loading states are handled, how record detail panels open and close, how status is communicated consistently. These are the decisions that, without a shared reference, each developer makes differently on every screen.
Staff experience the same behaviours on every screen. Learning one list means understanding all lists.

Ecosystem-wide DS audit
All 13 products
Applicaa's 13 products have grown independently. A school buying Admissions+ Post-16 and Options+ experiences two completely different visual languages. Common elements — headers, status badges, tables, filters — look and behave differently in each product. No product team can reuse what another has built.

This audit maps every product's screens against the DS component library. It identifies what each product can adopt immediately, what needs adapting, and what's genuinely unique to that product.
Shared header, navigation, and status patterns adopted by all 13 products. Every team knows what's in the DS and how to use it. Reuse replaces rebuild.
Shorter build time per productConsistent cross-product UXShared identity for Applicaa brand
🎨
Product identity system
All 13 products
Each product has its own colour personality — Exams+ gold, Options+ teal, MAT+ purple. Without rules, this is inconsistency. With rules, it becomes a feature: every product feels distinctly Applicaa while still being recognisably itself.

The DS defines a "brand layer": which visual tokens each product can customise (accent colour, product icon) and which are fixed across the whole family (spacing, typography, interaction patterns, accessibility standards).
Clear rules distinguishing shared standards from intentional product differentiation. Adopted first by Bursary+, Applicaa Futures, and Local Authorities — the products furthest from the current DS.
♿
WCAG 2.1 AA accessibility audit
All 13 products
UK public sector procurement requires WCAG 2.1 AA. MATs and Local Authorities are public sector — failing an audit risks losing contracts across the whole portfolio. Parent-facing products (Exams+, Bursary16+) must work for users with visual, motor, and cognitive impairments. Covers: text contrast, keyboard navigation, screen reader compatibility, focus management, form error announcements.
All critical findings fixed. Accessibility statement published for each product.
📱
Mobile and tablet — full pass
All 13 products
Parent-facing products (Exams+, Bursary16+, ApplicaaOne, Admissions+ for Early Years) are used primarily on phones — at school pickup, at home, on the bus. Phase 1 covered Admissions+ Post-16 critical screens. This pass covers every other product. Tested on real devices, not simulators.
Top 10 screens per product. Tables stack to card layout. Buttons meet minimum tap target size. Parent flows work thumb-only.
📖
DS contribution guide and governance
All 13 products
No documented contribution process. Bootstrap leakage in Admissions+ is the visible symptom — lack of governance is the cause. With 13 product teams contributing to a shared DS without rules, divergence grows exponentially. What Phase 0 fixed will return.
Published guide: naming conventions, documentation requirements, accessibility checklist, PR criteria. Automated linter blocks legacy classes from being merged by any team.
✨
Animation and motion standards
All 13 products
Transitions (panels, dialogs, notifications) vary in speed and style across all 13 products. A user switching between Admissions+ and Options+ on the same day experiences two different "feels". Four standard animation speeds defined and applied consistently. Respects reduced-motion system setting.
Consistent motion language across every product. Feels like one platform.

Phase 4 — AI-Native

Redesign how parents, students, and school staff interact with Applicaa — moving from forms to guided AI conversations. The biggest long-term opportunity for the platform: from software that makes admissions digital, to a service that actively guides families through it.
