Phase 0 — Shared Foundation

Establish the Admissions+ and MAT+ design foundation using the DS Lookbook — the single shared reference for how every component, pattern, colour, and interaction should look and behave. Every improvement in Phases 1 and 2 is built using this language.

What is the DS Lookbook?
The DS Lookbook at designsaplus.com/lookbook is the shared design system for Admissions+ and MAT+ — a living reference documenting every reusable component, shared pattern, and design token across both products. It is the source of truth for how things look and behave in these two products.

When a developer needs to build a new screen, they start here. When a designer proposes a change, they document it here first. When two screens look different, the Lookbook is the arbiter of which is correct.
What Phase 0 delivers
🧱
Component library
Admissions+
Every reusable UI element — inputs, buttons, tables, cards, panels, navigation, badges, and feedback components — defined in one place with a live preview, available variants, and usage guidance. No more ad-hoc rebuilds or searching for the right example.
Every developer and designer works from the same reference. Components look and behave consistently across every screen.
🎨
Design tokens
Admissions+
The colour palette, typography scale, spacing system, and status colour map — all named and published as shared tokens. Navy #263C91, Coral #EC634B, status colours (Upcoming = blue, Completed = green, Cancelled = red, Full = amber), and the full Inter type scale are all defined here.
Colour, type, and spacing are consistent across the product. A future rebrand or dark mode is a token change — not a product-wide rebuild.
📐
Shared patterns
Admissions+
Common interaction patterns documented beyond individual components — how filtering works across list screens, how empty and loading states are handled, how record detail panels open and close, how status is communicated consistently. These are the decisions that, without a shared reference, each developer makes differently on every screen.
Staff experience the same behaviours on every screen. Learning one list means understanding all lists.

Phase 1 — Quick Wins

Apply DS components to improve UI/UX across Settings, Processes, and Customised Areas — without redesigning screens yet. Each item is independently shippable and targets the highest-friction areas school staff encounter daily.

Settings
⚙️
Restructure settings navigation
Reorganise the 8 top-level sections into four audience-based groups. Add short descriptions to each group. Apply the DS active state pattern so the current page is always visible.
Staff find settings faster. Fewer support requests. Faster onboarding for new staff.
🔍
Settings search and wayfinding
Add a DS search input to the settings nav so staff can reach any page by name. Add breadcrumb context above each settings page to confirm where they are in the tree.
Any setting reached in seconds. Browsing the nav becomes optional.
Processes
📋
Application Form settings
Apply DS form group, section title, toggle, and label components to the form builder and field configuration screens. No redesign — component alignment only.
Form settings feel consistent with the rest of the product. Staff configure forms with less friction.
📨
Communications — Forms, Surveys, Templates
Apply DS rich text editor, preview panel, and status badge consistently across email templates, SMS templates, enquiry forms, and surveys. One editing experience across all four.
Staff learn the communication editor once and apply it everywhere. Fewer configuration errors.
📤
Import and export
Add DS progress and error components to the import flow. Add format selection and a download history panel to the export flow. Replace silent or technical feedback with plain language.
Staff know what is happening during imports and can repeat exports without support.
Customised Areas
🗓️
Timetable
Apply DS layout, form group, data table, and status badge to the timetable configuration screens. Aligns a high-traffic setup area with the rest of the product without rebuilding it.
Timetable setup uses familiar patterns. Fewer errors during initial school configuration.
🎫
Offer management
Apply DS status badges to offer rows, add a pipeline summary using the metric card component, and apply the bulk action bar for multi-row operations. No workflow changes — visual clarity only.
Offer states distinct at a glance. Pipeline totals visible. Bulk actions available without opening individual records.

Enrolment checklist
Apply DS card and progress components to introduce a per-student enrolment tracker. Completion percentage visible on the list view. Automated reminders for outstanding steps.
Staff can see enrolment progress at a glance. No manual follow-up needed for outstanding steps.

Ecosystem-wide DS audit
All 13 products
Applicaa's 13 products have grown independently. Admissions+ and MAT+ share the DS Lookbook. The remaining 11 products have grown their own visual patterns independently. A buyer using Admissions+ and Options+ experiences two completely different products.

This audit maps every product against the DS Lookbook. It identifies what each product can adopt immediately, what needs adapting, and what is genuinely unique to that product. The goal is a coherent family — not a single monolithic product.
Every product mapped to the DS Lookbook. Shared header, navigation, and status patterns adopted across the family. Consistent experience for schools using multiple Applicaa products.
Shorter build time per productConsistent cross-product UXShared identity for Applicaa brand
🎨
Product identity system
All 13 products
Each product has its own colour personality — Exams+ gold, Options+ teal, Bursary16+ green, MAT+ purple. Without rules, this is inconsistency. With rules, it becomes a feature: every product feels distinctly Applicaa while still being recognisably itself.

Phase 3 defines a brand layer on top of the DS Lookbook: which tokens each product can customise (accent colour, product icon) and which are fixed across the whole family (spacing, typography, interaction patterns, accessibility standards).
Clear rules distinguishing shared standards from intentional product differentiation. Adopted first by Bursary+, Applicaa Futures, and Local Authorities — the products furthest from the current DS.
♿
WCAG 2.1 AA accessibility audit
All 13 products
UK public sector procurement requires WCAG 2.1 AA. MATs and Local Authorities are public sector — failing an audit risks losing contracts across the whole portfolio. Parent-facing products (Exams+, Bursary16+) must work for users with visual, motor, and cognitive impairments. Covers: text contrast, keyboard navigation, screen reader compatibility, focus management, form error announcements.
All critical findings fixed. Accessibility statement published for each product.
📱
Mobile and tablet — full pass
All 13 products
UK schools are BYOD — iPads and Android tablets are primary devices for school staff. Parent-facing products (Exams+, Bursary16+, Admissions+ for Early Years) are used primarily on phones. Tested on real devices, not simulators.
Top 10 screens per product. Tables stack to card layout. Buttons meet minimum tap target size. Parent flows work thumb-only.
📖
DS contribution guide and governance
All 13 products
No documented contribution process exists for the DS Lookbook. Bootstrap leakage in Admissions+ is the visible symptom. With 13 products contributing to a shared DS without rules, divergence grows exponentially. What Phase 0 fixed will return.
Published guide: naming conventions, documentation requirements, accessibility checklist, PR criteria. Automated linter blocks legacy classes from being merged by any team.
✨
Animation and motion standards
All 13 products
Transitions (panels, dialogs, notifications) vary in speed and style across all 13 products. A user switching between Admissions+ and Options+ on the same day experiences two different "feels". Four standard animation speeds defined and applied consistently. Respects reduced-motion system setting.
Consistent motion language across every product. Feels like one platform.

Phase 4 — AI-Native

Redesign how parents, students, and school staff interact with Applicaa — moving from forms to guided AI conversations. The biggest long-term opportunity for the platform: from software that makes admissions digital, to a service that actively guides families through it.
