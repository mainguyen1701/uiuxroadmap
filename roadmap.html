March – April 2026 · 4 weeks

Design System Foundation

Fix the shared foundation before building on it. Broken previews, token leakage, and semantic errors compound across every screen they touch. These must be resolved first.

8
initiatives
4
P0 items
Repair — Fix what's broken before building
Publish 4 broken Lookbook previews
P0MDS
Problem: DataTable, TagInput, InfoGrid, PersonCard are in the nav but return 404. They exist in code but are invisible to the team — so everyone rebuilds them ad-hoc.
Write Lookbook previews for all 4. Add usage notes, prop docs, and at least one real ApplicaaOne data example per component.
Remove Bootstrap class leakage
P0SDS
Problem: 14+ instances of fw-medium, text-muted, fw-semibold mixed into tw- component templates. If Bootstrap is removed, these silently break.
Find-replace in ViewComponent templates: text-muted → tw:text-neutral-400, fw-medium → tw:font-medium, fw-semibold → tw:font-semibold.
Fix semantic badge colours
P0SDS
Problem: "Upcoming" events use tw-badge-success-subtle (green). Green = completed/good. An upcoming event is a neutral future state — it reads as if the event already succeeded.
Add tw-badge-info-subtle (blue). Map: Upcoming → info, Completed → success, Cancelled → danger, Full → warning, Draft → neutral.
Reduce table column density
P0MMAT+
Problem: Events table has 15 columns. Overflows on laptops, unusable on iPads (the primary BYOD device in UK schools).
Default to 7 columns (Actions, Date, Event, RSVPs/Cap, Status, Source, Tickets Remaining). Add column chooser toggle using existing view_toggle_controller.
Build — New P0 components
Empty State component
P0MAll
Problem: Every table, list, and panel handles zero-data ad-hoc with inline text-muted + icon. No consistent empty pattern exists across the 3 products.
Build tw-empty-state with slots: illustration/icon, heading, body, CTA button. Variants: no-data, no-results (after filter), first-use (onboarding prompt).
Skeleton Loader component
P0MAll
Problem: Async data fetches (events table, RSVP list, dashboard KPIs, Bromcom import results) show nothing while loading. Causes layout shift and feels broken.
Build tw-skeleton with shimmer animation. Variants: text line, avatar, card, table row, stat block. Uses CSS animation — no JS dependency.
Filter Chips component
P0MAll
Problem: Filter chips are used in Push to Schools, events list, school phase filters, and year group filters — but built as bespoke Alpine.js each time with no shared component.
Build tw-filter-chip and tw-filter-chip-group. States: default, active, disabled. Optional count badge, clear-all shortcut, "All" default chip.
Date & Time Picker
P0LAll
Problem: An events platform without a designed date picker falls back to native <input type="date"> — which renders differently on every browser and doesn't match the design language.
Build tw-date-picker and tw-time-picker. Modes: single date, date range, time-only. Integrates with Stimulus and form validation. Keyboard accessible.

May 2026 · 4 weeks

Quick Wins — All Products

High-impact, low-effort fixes that improve the daily experience for school staff users and shift the platform perception. Each item can be shipped independently.

10
initiatives
5
P0/P1 items
ApplicaaOne — Marketing site (scored 4.4/10 vs 8.4 benchmark)
Value proposition above the fold
P0SApplicaaOne
Problem: Homepage hero shows no clear statement of what ApplicaaOne does or who it's for before the scroll. Same H1 tag used on every page — zero SEO differentiation.
Rewrite hero: unique H1 per page, one-sentence value prop visible within 300px viewport height, primary CTA above fold. Add school type selector (secondary / post-16 / primary).
Trust layer — reviews & social proof
P0MApplicaaOne
Problem: 105 reviews listed for 45,000+ courses. 2 homepage reviews read as auto-generated ("BEST SCHOOL", "Is good"). Destroys trust at the point of decision.
Gate reviews with verified post-event confirmation email. Remove reviews below 20 words. Surface verified count ("4,200 verified reviews") as a stat, not individual low-quality text.
Navigation overhaul
P0MApplicaaOne
Problem: Navigation doesn't separate student-facing from school-facing journeys. A school staff member and a Year 10 student hit the same nav — different needs, one confusing structure.
Split nav into two clear paths: "For Students" and "For Schools". Persistent search bar for course discovery. Mobile-first hamburger menu with clear back-navigation.
Colour token consolidation
P1SApplicaaOne
Problem: 7 different blue class variants in use: text-primary-5, text-brand-blue, text-blue-5, text-blue-600, bg-[#2D8FC5], bg-[#1E2F72], bg-blue-500. Impossible to maintain.
Consolidate to 3 semantic tokens: brand-primary (#263C91), brand-secondary (#2D8FC5), brand-dark (#1E2F72). Alias all existing classes to these three.
Admissions+ — Core platform
Typography consistency — Inter rollout
P1SAdmissions+
Problem: Admissions+ currently uses Proxima Nova + Arial — $199/yr licensing cost, 150–200KB payload. No variable font support. Not aligned with MAT+ design system which recommends Inter.
Replace with Inter (open-source, UI-optimised, ~50–80KB). Update CSS variable: --font-body: 'Inter', system-ui. Test with school admin screens — dense data tables benefit from Inter's narrow numerals.
Mobile responsiveness audit — core flows
P1MAdmissions+
Problem: UK schools use iPads and Android tablets (BYOD) as primary devices. Admissions+ was designed desktop-first — table-heavy layouts break below 768px.
Audit 5 highest-traffic flows (application form, offer management, enquiry list, event bookings, student dashboard) on 768px and 375px viewports. Stack tables to card layout on mobile.
MAT+ — Platform improvements
Move QR codes out of table rows
P1SMAT+
Problem: Each events table row renders a 64×64px QR code + Download button inline in the Event cell, forcing 100px+ row height. Scanning a list of 20+ events is painful.
Move QR access to the row action dropdown menu as "Download QR Code". Table rows return to single-line height (~40px). QR shown in a popup/modal on click.
Page-level system banner
P1MAll
Problem: No component exists for platform-wide persistent alerts (Bromcom import in progress, subscription expiring, maintenance window, MIS sync delay). These are currently unhandled.
Build tw-banner — sits above the topbar, full-width, dismissable. Variants: info, warning, critical. Managed server-side via Rails flash or a dedicated Banner model.

June – July 2026 · 8 weeks

Core Component Build

Build the missing P1 components that gate major product features. These unblock the Phase 3 product revamp — without them, teams build custom one-offs in every screen.

7
components
6
P1 items
Detail Slide-out Panel
P1LAll
Problem: The RSVP panel, school detail view, and student record all slide in from the right — built custom each time. The nav Drawer is a navigation component, not a detail panel.
Build tw-panel: right-side slide-in with sticky header, scrollable body, sticky footer (actions). Sizes: sm (480px), md (600px), lg (800px). Backscroll lock, close-on-backdrop, focus trap.
Bulk Action Bar
P1MAll
Problem: Every table with bulk-select re-implements the selection state toolbar (X items selected | Publish | Cancel | Delete). No shared component — inconsistent placement and behaviour.
Build tw-bulk-bar: contextual bar that appears above/within the table on checkbox selection. Slots for count, primary action, secondary actions, deselect-all. Connects to filter_controller.
Stat / KPI Card
P1MDS
Problem: infogrid is in the Lookbook nav but returns 404. The Insights dashboard, MAT+ overview, and Admissions+ reports all need metric cards with trend indicators — currently all custom.
Build and publish tw-stat-card: primary metric (large), label, delta indicator (▲▼ with colour), comparison text ("vs last month"), optional sparkline slot. Works standalone and in grid.
Combobox / Multi-select
P1LAll
Problem: School selection, year group selection, staff assignment, and subject-level filtering all need searchable multi-select. The existing Dropdown and FormSelect don't support search + constrained multi-select.
Build tw-combobox: keyboard-navigable searchable select. Supports single + multi-select mode, option groups, "select all", remove chips on selection. Stimulus-powered, no external dependency.
Calendar View
P1XLMAT+Admissions+
Problem: The events calendar in MAT+ is entirely custom Alpine.js — not a DS component. Can't be reused for Admissions+ event bookings or the school calendar without full re-implementation.
Build tw-calendar: month + week view toggle, event pill in cell (colour by type), today highlight, click-to-open popup hook. Renders from a data attribute — server renders events as JSON. Stimulus controller.
Avatar Group
P2SAll
Problem: Event attendees, school staff lists, and team member displays all need overlapping avatar stacks. Currently shown as count text with no visual identity.
Build tw-avatar-group extending existing tw-avatar. Max visible: 3 (configurable). Overflow: "+N more" tooltip showing full list. Variants: sm/default/lg.
Split Button
P2MMAT+
Problem: Publish, Push to Schools, and Make Offer actions all have a primary action + secondary options — currently shown as separate Button + Dropdown pairs with no shared housing or consistent placement.
Build tw-split-btn: primary action area + chevron dropdown trigger as distinct click targets, sharing the same button height and border radius. Variants: primary, success. Dropdown inherits dropdown_controller.

Product-Specific Revamp

Major screen and flow redesigns for each product, using the DS components built in Phases 0–2. Each product has its own sprint lane — they can run in parallel.

12
initiatives
3
products
Admissions+ — Core flow redesign
Application form — multi-step redesign
P0XLAdmissions+
Problem: Application forms are long, single-page scrolls. Parents on mobile abandon multi-section forms with no sense of progress. No save-and-return pattern.
Redesign as guided multi-step flow using tw-steps. Each section = one step (Personal details, Parent details, Additional info, Review & submit). Auto-save on step transition. Progress bar persistent in header.
Parent-facing status tracking
P0LAdmissions+
Problem: Parents have no clear view of where their application is in the process. Results in high support volume ("Has my application been received?") and poor NPS scores.
Build application status dashboard: tw-steps showing current stage (Submitted → Under Review → Offer Made → Accepted/Declined). Status badge, last-updated timestamp, next action CTA.
Offer management UI
P1LAdmissions+
Problem: Making and managing offers — conditional, unconditional, waitlisted, declined — involves dense admin tables with no visual differentiation between offer states.
Redesign offer management screen: KPI cards at top (offers made/accepted/declined), status-coloured table rows, bulk offer actions using tw-bulk-bar, offer detail in tw-panel slide-out.
Event booking — redesign for schools
P1LAdmissions+
Problem: Event booking in Admissions+ is separate from MAT+ Events. The two surfaces are visually inconsistent — school staff manage events differently depending on which product they're in.
Unify event booking UI using the tw-calendar and Events DataTable built in Phase 2. Consistent action model: Add → Push → Manage RSVPs → Close. Use shared DS components across both products.
MAT+ — Dashboard & oversight
MAT dashboard — KPI overview
P0XLMAT+
Problem: MAT leaders need a single-pane view of all schools — total applications, offers made, acceptance rate, event attendance — but the current overview is list-based with no metrics.
Redesign MAT dashboard: top row of tw-stat-card KPIs (network-wide metrics), school comparison table sorted by key metric, inline trend indicators, drill-down to school via tw-panel.
Multi-school permissions model — clarity
P1LMAT+
Problem: The current permissions UI (staff members panel) is a modal with inline access-level dropdowns. Adding/removing staff, understanding what each level can do, and seeing cross-school access is difficult.
Redesign as a dedicated Permissions screen (not modal). School-grouped staff list, access level legend table, add staff flow using tw-combobox, changes staged and saved in bulk.
Bromcom import validation — progress UI
P1MMAT+
Problem: Bromcom data imports run silently with no visible progress. Staff don't know if the import succeeded, is still running, or failed until they navigate away and back.
Add import progress using tw-progress (indeterminate while running, determinate on validation pass). Success/error state using tw-inline-notification. Error rows surfaced in a diff-style table.
ApplicaaOne — Student & school journey
Course discovery redesign
P0XLApplicaaOne
Problem: 45,000+ courses with poor discoverability. No subject grouping, no school comparison, no "courses near me" proximity filter. Students land on a flat list and abandon.
Redesign course search: subject taxonomy filter (A-level, BTEC, T-Level, IB), proximity radius slider, school comparison shortlist (max 3), saved courses. Filter state persists on back-navigation.
School profile page
P1LApplicaaOne
Problem: School pages are generic listing pages with no personality or trust signals. A student choosing between two sixth forms needs to understand culture, results, transport, and open days — none of which is surfaced.
Redesign school profile: photo hero, key stats row (Ofsted rating, results, distance), course list with subject filter, upcoming open events, verified reviews section, sticky "Apply / Book a Visit" CTA.
Student registration flow
P1LApplicaaOne
Problem: Registration is a blocker before applying or booking events. High drop-off at account creation. Form is not mobile-optimised and collects more fields than needed at this stage.
Streamline to 3 fields at registration (name, email, school year). Defer additional details to the application form. Add social/SSO login (Google, Microsoft — used by most schools). Progress saved immediately.

November – December 2026 · 8 weeks

Scale & Polish

Platform-wide quality pass: accessibility, mobile, animation, and documentation. These ensure the revamp holds long-term and the DS is maintainable by the full team.

6
initiatives
WCAG 2.1 AA accessibility audit
P0LAll
Problem: UK public sector procurement increasingly requires WCAG 2.1 AA compliance. Schools and MATs are public-sector bodies — Applicaa touching student data must meet accessibility standards.
Audit all 3 products against WCAG 2.1 AA: colour contrast (4.5:1 text, 3:1 UI), keyboard nav, focus management, screen reader labels, skip links, form error announcements. Fix critical findings. Publish accessibility statement.
Mobile-first responsive pass — product
P1LAll
Problem: Phase 1 audited Admissions+ on mobile. This pass covers MAT+ and ApplicaaOne product screens — the 10 highest-traffic views per product on 375px and 768px.
Responsive audit + fixes for all three products. Convert remaining desktop-only tables to card layout on mobile. Test on real iOS Safari and Android Chrome — not just Chrome DevTools.
Micro-interactions & animation tokens
P2MDS
Problem: Transitions between states (modal open, panel slide, skeleton → content, toast appear) are inconsistent. Some use Tailwind transitions, some Stimulus, some CSS keyframes — no shared motion language.
Define 4 motion tokens: duration-fast (100ms), duration-base (200ms), duration-slow (350ms), easing-standard (ease-in-out). Document per-component. Respect prefers-reduced-motion.
DS contribution guide & governance
P1MDS
Problem: No documented contribution process for the design system. Any developer can add a component in any style — which is how Bootstrap leakage and semantic colour misuse happened in the first place.
Publish contribution guide in Lookbook: component API contract, naming conventions, required Lookbook scenarios, accessibility checklist, PR review criteria. Add automated linting for Bootstrap remnant classes.
Dark mode token preparation
P2MDS
Problem: No dark mode exists. As school admin tools increasingly run on personal devices, dark mode is a growing expectation — especially for evening school events and late-night parent registrations.
Define semantic colour tokens (not raw hex): surface-primary, surface-secondary, text-primary, text-secondary, border-default. Map to light values now; dark values later. No UI switching in this phase.
Diff view component
P2MMAT+
Problem: The school edits comparison (MAT original vs school version, changed rows highlighted amber) is fully custom in the Events prototype. The same pattern is needed for Forms, Policies, and Offers.
Extract into tw-diff-table: three-column layout (field, original value, changed value), changed rows highlighted, unchanged rows dimmed. Data-driven from a JSON diff payload. Reusable across all modules.
